## 1. 引言

`/loop` 这个东西挺好用,日常用 CodeBuddy、Claude Code 这种本地 Agent,我经常用 `/loop` 搞个每日数据统计、日志巡检之类的活。

但有个坑第一次用就踩到了:**`/loop` 创建的定时任务,固定 72 小时自动销毁**,到了时间点直接消失,任务就这么没了。

这文章就把考虑过的路子、哪些是假动作、哪些真能用,都摆出来。

---

## 2. 72 小时过期到底是什么机制

### 2.1. 两个时间戳

每条 `/loop` 任务在创建的时候,后端会写两个时间戳:

- `create_ts`:任务创建那一刻,固化下来,谁也动不了
- `expire_ts = create_ts + 72h`

Agent 后台有个巡检线程一直在跑,看到 `expire_ts` 到了,直接把这个 loop 任务**强制删除**。

### 2.2. 之前踩的几个误区

一开始我以为"会话活跃就能续命",所以试了好几种办法,全没用:

1. 没事就给 Agent 发消息,让它别闲着 → 没用,过期时间戳不动
2. 反复重启 WebUI、Daemon 进程 → 没用,`create_ts` 不变
3. 想改本地缓存文件手动改时间 → 没用,有内存缓存 + 哈希校验,改了直接被覆盖回来
4. 找了一圈官方文档和命令,没有任何"刷新 TTL"的指令

**一句话总结**:`/loop` 的过期是**固定生命周期销毁**,不是"闲置超时销毁"。你就算 24 小时跟 Agent 聊天打屁,到点照样炸。

---

## 3. 我试过的几种"续命"招,全废了

这一节就是我折腾的过程,记录下来给同样踩过坑的人一个参考。

### 3.1. 单 Loop 单独跑

这不废话嘛,到点就死。

### 3.2. 双 Loop 互相续期

想了个"聪明"的办法:A 是业务 loop,B 是定时去重建 A 的 loop。

听起来挺好?但 B 它**本身也是个 loop**,也是 72 小时寿命。B 一死,A 没人续,整套还是断。这只是把"3 天断"延长到了"6 天断",本质问题没解决。

### 3.3. 三 Loop 环形互保

又试了个更花哨的:三个 loop 互相重建,故意把创建时间错开,降低同时过期的概率。

听着挺美,实操下来问题一堆:Daemon 重启、网络抽风、某次执行失败,这几个时间轴就慢慢对齐了,然后**三个一起炸**。这种"概率续命"压根不能上生产。

### 3.4. 频繁互动保活

这招最朴素,也没用。会话活不活跃,跟 loop 任务的生命周期一点关系都没有。

---

## 4. 真正能用的两条路

能绕开 72 小时限制的,有两条路:

1. **用 Goal 续 Loop**(借 Agent 内部的常驻机制,适合个人临时用)
2. **彻底弃用 Agent 调度,改用系统定时**(适合长期稳定跑)

### 4.1. Goal 续 Loop

#### 4.1.1. 为什么 Goal 能行

| 指令    | 它是干啥的   | 过期机制                |
| ------- | ------------ | ----------------------- |
| `/loop` | 定时调度任务 | 72 小时 TTL,到点强制删  |
| `/goal` | 常驻目标     | 没过期时间,跟会话同生死 |

所以思路很简单:搞一个**永久不死的 Goal**,让它每天去把所有快过期的 Loop 续一下。

#### 4.1.2. 我用的 Goal 指令

我自己的 Goal 是这么写的,贴出来给你参考:

```text
/goal 每24小时执行一次全局loop保活巡检，持续永久运行：
1. 列出当前会话所有loop任务、创建时间、剩余有效期、完整执行指令
2. 筛选剩余有效期小于24小时的即将过期任务
3. 删除旧的即将过期loop
4. 严格复刻原指令重新创建loop，刷新生命周期时间戳
5. 将本次续期结果、任务清单写入本地日志文件留存
```

#### 4.1.3. Goal 这条路其实挺脆的

我用了大概一个月,中间就翻车过两次,后来就知道这玩意不能完全依赖:

1. 我手贱敲了一次 `/clear`,Goal 直接没了,Loop 也没人续了
2. CodeBuddy Daemon 进程被系统升级重启了,Goal 不会自动恢复
3. 会话异常中断 + 重建,Goal 也彻底没了

**一旦 Goal 没了,3 天后所有 Loop 集体停摆**,而且不会给你发通知那种停摆,完全静默。

所以这条路**只适合个人临时测试、几周内的短期任务**。要拿它当生产方案,迟早出事。

### 4.2. Windows 任务计划 + CodeBuddy CLI 外置调度

这条路是**彻底不依赖 Agent 内置调度**,把定时这事交给操作系统。

#### 4.2.1. 核心思路

- Windows 任务计划:系统级的东西,没 TTL 概念,可以开机自启、崩了自愈
- CodeBuddy CLI 退化成"执行工具",不负责调度,只负责干活

#### 4.2.2. 最简落地

我自己跑下来的最小化结构就三件东西:

1. `task.txt`:扔一些任务指令文本
2. 一个 bat 脚本:指定 Session + 读 task.txt + 调 CLI
3. Windows 任务计划:按时触发 bat

#### 4.2.3. bat 脚本示例

```bat
@echo off
chcp 65001
:: 替换为你的真实会话ID
set SESSION_ID=xxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxx
:: 任务指令文件路径
set TASK_TXT=D:\codebuddy_task\task.txt
:: 日志路径
set LOG=D:\codebuddy_task\run_log.txt

codebuddy -r %SESSION_ID% -p -y @%TASK_TXT% >> %LOG% 2>&1
```

里面几个参数是我自己试出来的:

| 参数 | 干啥用的                              |
| ---- | ------------------------------------- |
| `-r` | resume 固定 Session,上下文不丢        |
| `-p` | 静默执行,不弹交互框                   |
| `-y` | 权限弹窗自动放行,定时触发时不会被卡住 |

#### 4.2.4. Windows 任务计划配置(精简版)

1. 打开「任务计划程序」→ 创建基本任务
2. 触发器按你想要的周期(每天/每周) → 下一步
3. 操作选"启动程序",选你那个 bat
4. 勾上「不管用户是否登录都运行」→ 完成
5. 想要开机自启:任务属性里勾「唤醒此计算机运行此任务」

#### 4.2.5. 这条路的好处

跑下来大半年没翻车,几个核心优点:

- 完全不受 3 天 loop 过期影响
- 会话重启、Daemon 重启、`/clear` 都不影响任务本身
- `-r` 复用 Session,业务上下文一直在
- 真正无人值守,跑几个月不用管

## 5. 怎么选

我自己现在的选法,基本就是看场景:

- 临时测试、不想折腾系统配置 → Goal 续 Loop(但心里清楚会翻车)
- 长期业务、生产环境、要稳 → Windows 任务计划 + CLI(就是配起来稍微麻烦点)

Goal 那条路,最大的坑是**翻车是静默的**。Goal 没了,Loop 不会给你发个通知"我要死了",就是 3 天后默默停摆,等你某天突然发现报表没更新才反应过来。

Windows 外置这条路,虽然配起来麻烦点(第一次得折腾 bat 和任务计划),但跑起来之后基本可以忘了它的存在。**真正脱离 Agent 引擎所有生命周期限制**的方案,目前看也就这一条。

---

## 6. 总结

最后总结就三句话:

1. `/loop` 的 72 小时过期是引擎硬编码,别想着靠"互动保活"或者"改文件"绕过
2. Loop 互相续期那些花活都是假动作,本质问题没解决
3. 个人短期用 Goal 续一续,长期跑业务就老老实实上系统定时
